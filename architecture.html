Shared State — AgentState

Every node reads from and writes to a single AgentState TypedDict. LangGraph passes the state through the graph, so nodes communicate purely through these fields — no global variables, no side channels.

from typing import TypedDict, Any, Literal class AgentState(TypedDict): question: str # raw user input intent: Literal["pl", "property", "compare", "general", "clarify"] filters: dict # {property_name, tenant_name, year, quarter, month, ledger_type} data: Any # list[dict] or {prop: profit} — filled by worker nodes answer: str # final natural-language response error: str # propagated to formatter if something fails
Component Deep Dive
🧠
Supervisor Node
nodes/supervisor.py

The entry point. Makes one GPT-4o call with all canonical values from the CSV injected into the system prompt, returning a JSON with intent and filters.

  • Injects known properties, tenants, ledger types into prompt
  • Forces JSON mode for reliable parsing
  • For compare intent: runs llm_resolve as fallback if property names don't match
📊
P&L Node
nodes/pl.py

Handles profit & loss queries. Calls query_pl() which groups by property_name, month, ledger_type, ledger_group and sums profit.

  • Filters applied before groupby
  • Gracefully returns empty answer if no data
  • Passes rows to formatter
🏠
Property Node
nodes/property.py

Returns raw filtered rows from the CSV — useful for tenant-level drill-down, month-by-month views, or specific ledger inspection.

  • Calls query_property() — no aggregation
  • Returns df.to_dict("records")
  • Streamlit shows top 20 rows in a table
⚖️
Compare Node
nodes/compare.py

Side-by-side property comparison. Requires at least 2 resolved property names, then sums profit per property.

  • Validates against canonical list first
  • Double fallback: llm_resolve on unmatched names, then on full question
  • Returns {"Building 180": 95400, …}
💬
General Node
nodes/general.py

Handles real estate knowledge questions (cap rate, NOI, market trends, investment advice) that don't require querying the dataset.

  • GPT-4o acts as a domain expert
  • Skips formatter — goes straight to END
  • Never references internal data
❓
Clarify Node
nodes/clarify.py

Handles truly ambiguous queries. GPT-4o asks exactly one targeted question to help the user refine their request.

  • Suggests known property names & years in the prompt
  • Never attempts to answer the question
  • Used sparingly — supervisor avoids it if intent is inferrable
✍️
Formatter Node
nodes/formatter.py

Turns raw data rows into a 2–6 sentence human-readable answer. Enforces consistent money formatting and concise style.

  • Receives question + data
  • Post-processes with regex to fix spacing artifacts
  • Shared by pl, property, and compare nodes
🔧
LLM Resolve Tool
tools/llm_resolve.py

Maps informal property references (nicknames, addresses, partial names) to canonical names from the CSV. Only returns names that exist verbatim in the known list.

  • GPT-4o with JSON mode
  • Extra validation pass after parsing
  • Used by supervisor and compare node
📂
Data Tool
tools/data.py

CSV loaded once at module import into a global DataFrame. All queries filter this in-memory DataFrame — no database, no repeated I/O.

  • get_unique_values(col) — for prompt injection
  • query_pl(filters) — grouped P&L
  • query_property(filters) — raw rows
  • query_compare(props, filters) — profit per property
🌐
Streamlit App
app.py

Three-column layout: guided input on the left, chat window in the center, diagnostics panel on the right.

  • Left: dropdowns to build structured questions
  • Center: scrollable chat with inline metrics tables
  • Right: shows intent, filters, row count for each response
🧪
Eval Harness
eval.py

Root-level evaluation script that runs defined test cases through the compiled graph and reports accuracy metrics.

  • Checks intent accuracy
  • Checks answer non-empty rate
  • Checks numeric match rate in answers
  • Structured failure report per case
Data Flow Walkthrough — Example: "Compare Building 180 vs Building 140 in 2025"
1

User submits question via Streamlit

The form in app.py captures the text and calls graph.invoke({"question": ..., ...}) with a fresh initial state.

2

Supervisor makes GPT-4o call

The system prompt includes all canonical property names, tenants, years, and quarters pulled from the live CSV. GPT-4o returns {"intent": "compare", "filters": {"properties": ["Building 180", "Building 140"], "year": "2025"}}.

3

Supervisor post-validates property names

Each extracted name is checked against get_unique_values("property_name"). Any unmatched names are sent to llm_resolve_properties() for a second resolution attempt. The validated list is saved to state["filters"]["properties"].

4

Router sends state to Compare Node

LangGraph's conditional edge reads state["intent"] and routes to the compare node.

5

Compare Node queries the DataFrame

query_compare(["Building 180", "Building 140"], {"year": "2025"}) filters the in-memory Pandas DataFrame for each property and sums the profit column. Returns {"Building 180": 95400.0, "Building 140": 82100.0}.

6

Formatter Node narrates the data

GPT-4o receives the question and the raw dict. It produces: "Building 180 generated higher net income than Building 140 in 2025: $95,400 vs $82,100." A regex cleanup pass removes any spacing artifacts.

7

Answer returned to UI

state["answer"] is displayed in the chat window. The diagnostics panel shows intent: compare, the filters, and the row count used. Streamlit renders a side-by-side metric widget for each property profit.

Key Design Decisions
🔒

JSON-mode everywhere

All GPT-4o calls that need structured output use response_format={"type": "json_object"}, making parsing deterministic and eliminating markdown-wrapped JSON bugs.

📌

Canonical values injected into prompts

Property names, tenants, years, and quarters from the live CSV are injected into the supervisor's system prompt on every call. This keeps the LLM grounded and prevents hallucinated filter values.

🔄

LLM-based property resolution

Instead of pure fuzzy matching, llm_resolve_properties() uses GPT-4o to map any informal reference (address, nickname, partial name) to canonical names — more robust for real-world inputs.

💾

Module-level DataFrame

The CSV is loaded once at import time into a global _df. All queries operate on in-memory Pandas — zero database setup, near-instant filter operations for a ~4k row dataset.

✂️

Formatter as a shared output layer

Separating data retrieval (pl/property/compare nodes) from narration (formatter) keeps each node small and testable. The formatter has one job: turn raw numbers into prose.

🧪

Eval harness from day one

eval.py provides intent accuracy, answer non-empty rate, and numeric match rate across defined test cases. This enables regression testing when the prompt or routing logic changes.

🚦

Clarify intent used sparingly

The supervisor prompt explicitly instructs GPT-4o to prefer a real intent over "clarify" — even when property names are unrecognized. Clarify is a last resort, not the default for partial matches.

⚡

No async, no streaming, no retries

The system is intentionally simple: synchronous calls, no complex retry logic. This makes the code easy to read, debug, and extend — appropriate for a portfolio/demo-scale project.
